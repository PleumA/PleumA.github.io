BUG REPORT: When isCustomDateRange is true and the range spans 
across a month boundary (e.g. 22 Jul – 31 Aug), the Calendar view 
and Person view display day labels as a continuously incrementing 
index (e.g. ...30, 31, 32, 33... 41) instead of resetting at each 
month boundary and showing the actual date. A label like "32 Sat" 
or "41 Mon" does not correspond to any real calendar date and is 
confusing to users.

ROOT CAUSE TO INVESTIGATE: Find where day labels are generated 
in custom range mode. It's likely using a simple loop counter 
(dayIndex + startOffset) instead of reading the actual day-of-month 
and month name from scheduleDates[dayIndex] (a native Date object).

REQUIRED FIX:

1. In renderCalendarView(), when isCustomDateRange is true, each 
   day cell's displayed number must be 
   scheduleDates[dayIndex].getDate() (the real day-of-month, 
   1-31, resetting each month) — not a running index.

2. When a custom range crosses a month boundary, add a visible 
   month label or divider so the user can tell where one month 
   ends and the next begins (e.g. a small "August" header above 
   the first cell of August, or include the month abbreviation 
   next to the day number for the first day of each new month, 
   such as "1 Aug").

3. In the Person view (pivoted grid), column headers must show 
   the actual day-of-month and weekday from scheduleDates[dayIndex], 
   not a sequential counter. Same month-crossing clarity applies — 
   if the row of dates crosses into a new month, indicate it 
   clearly (e.g. a thin divider line between Jul 31 and Aug 1, 
   or repeat the month abbreviation at each month's first column).

4. In renderTableView() (List view), confirm dates are already 
   displaying correctly per the original Custom Date Range 
   implementation (e.g. "22 Jul") — if List view is correct but 
   Calendar/Person are not, this confirms the bug is isolated to 
   those two specific renderers using a different (incorrect) 
   labeling method than List view.

5. Do NOT change anything for standard Month/Year mode 
   (isCustomDateRange = false) — this bug fix only applies to 
   custom range rendering paths.

6. add change event listeners to both inputStartDate and inputEndDate. Now, as soon as a user selects or types a date, the app checks if both fields have valid inputs, and if they do, it automatically triggers generateSchedule().
